    Create the runtime API client infrastructure with retry/fallback, plus the P-08 frontend page.

    **Backend:**

    1. Create `ApiClient.java` interface:
       - `&lt;T&gt; ApiResponse&lt;T&gt; execute(String configCode, RequestSpec request)`:
         - Takes a config code (looks up DB config) and a request specification
         - Returns typed response or throws ApiException
       - `&lt;T&gt; ApiResponse&lt;T&gt; executeWithFallback(String configCode, RequestSpec request, FallbackHandler&lt;T&gt; fallback)`:
         - Same as execute but with fallback handler for graceful degradation
       - RequestSpec: inner class with method (GET/POST/PUT), path (relative URL appended to endpoint), headers (Map), queryParams (Map), body (Object)
       - ApiResponse: inner class with success (Boolean), data (T), statusCode (int), responseTimeMs (long)

    2. Create `FallbackHandler.java`:
       - @FunctionalInterface
       - `ApiResponse&lt;T&gt; handle(ApiException exception)`: return a fallback response when the API call fails
       - Default implementations:
         - `static &lt;T&gt; FallbackHandler&lt;T&gt; emptyResponse()`: returns ApiResponse with success=false, data=null
         - `static &lt;T&gt; FallbackHandler&lt;T&gt; defaultData(T defaultData)`: returns ApiResponse with success=false, data=defaultData
         - `static &lt;T&gt; FallbackHandler&lt;T&gt; fromCache(String cacheKey, Class&lt;T&gt; type)`: returns cached data if available
       - All fallbacks log the exception via SLF4J

    3. Create `ApiClientFactory.java`:
       - @Component
       - Injected: ApiConfigService, RestTemplate, RetryRegistry
       - `ApiClient createClient(String configCode)`:
         - Load ApiConfigEntity by configCode
         - If not found or status=0: throw ApiConfigNotFoundException
         - Create RestTemplate with per-config timeouts (from config fields)
         - Create Resilience4j Retry object with per-config settings (retryCount, retryInterval, backoffStrategy)
         - Return RetryableApiClientImpl wrapping the RestTemplate + Retry + config
       - Cache created clients in a ConcurrentHashMap (configCode -> ApiClient)
       - `evictClient(String configCode)`: remove from cache (called when config is updated)
       - `evictAll()`: clear cache (called via refreshCache)

    4. Create `RetryableApiClientImpl.java`:
       - Implements ApiClient
       - Constructor takes: RestTemplate, Retry (from Resilience4j), ApiConfigEntity (for endpoint URL construction)
       - `execute(configCode, request)`:
         - Not used directly (client is already bound to a config)
         - Actually: the client wraps a specific config, so execute() uses its pre-configured RestTemplate and Retry
         - Call: `Retry.decorateSupplier(retry, () -> doHttpCall(request)).get()`
         - measure response time, wrap in ApiResponse
       - `executeWithFallback(configCode, request, fallback)`:
         - Try execute()
         - On ApiException: call fallback.handle(exception)
       - `doHttpCall(RequestSpec request)`:
         - Build full URL: {endpointUrl}{path}
         - Set headers based on authType:
           - NONE: no additional auth headers
           - API_KEY: add X-API-Key header with apiKey
           - BEARER: add Authorization: Bearer {apiKey}
           - BASIC: add Authorization: Basic Base64(apiKey:secretKey)
         - Execute with RestTemplate.exchange()
         - Parse response, return data
       - All HTTP calls use the per-config connectTimeout and readTimeout
